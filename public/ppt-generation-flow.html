PPT 不是一步生成的，而是一条受控流水线

直白说，系统先弄清楚“这节课是什么”，再找教学证据，接着规划整套课件，最后逐页生成、检查、修复和导出。

一句话版

老师需求先变成课程上下文和任务包；RAG 找到证据；PPT Agent 先生成大纲，再生成每页内容； 系统把页面内容转成 Slide IR，再渲染成 Web 预览和 PPTX。

Slide IR 可以理解为 PPT 的中间图纸：里面写清楚文本框、图片、形状、公式放在哪里。

1

收集课程信息

从老师对话、上传资料和页面状态里提取学科、年级、主题、课时、教学目标、重点难点。

产物：fields
2

确认课程大纲

先生成这节课的结构：导入、概念讲解、例题、练习、总结。老师确认后，下游才生成 PPT。

产物：lessonBlueprint
3

检索教学证据

RAG 从教材、历史 PPT、题库、学生作答和错因分析里找依据，不让模型凭空生成。

产物：Evidence Pack
4

规划整套 PPT

PPT Agent 决定整套课件有几页，每页承担什么教学任务，哪些页讲概念，哪些页做练习。

产物：deck plan / outline
5

逐页生成内容

系统按大纲逐页生成标题、正文、公式、例题、互动提示和讲稿，不是一次性吐出整个文件。

产物：slide draft
6

转成页面图纸

把每页内容转成结构化页面元素，写清楚文本框、图片、形状、公式的位置和样式。

产物：Slide IR
7

渲染与检查

用同一份页面图纸生成 Web 预览和 PPTX；检查文字是否超框、元素是否重叠、页面是否过满。

产物：WebSlide / quality report
8

修复并导出

发现问题就缩短文字、换布局、调整字号或降级稳定模板；最后导出 PPTX，也可继续在平台编辑。

产物：PPTX / editable state

每一步谁负责

Harness 管任务边界、上下文、工具权限和质量门禁，防止模型乱改状态。
RAG Agent 负责找证据，把教材、题库、学情等结果整理成 Evidence Pack。
PPT Agent 负责规划整套课件和逐页生成内容。
Compiler 把内容变成可计算的页面元素，再生成 Web 预览和 PPTX。
Critic 检查证据、内容、版式和导出风险，必要时触发修复。

为什么要这么多步

  • 先确认课程大纲，避免 PPT、题目、教案各讲各的。
  • 先做 RAG 取证，降低无依据生成和学科幻觉。
  • 先生成页面结构，再生成视觉元素，方便检查和修复。
  • Web 预览和 PPTX 共用一份结构化数据，减少两边不一致。
  • 导出前做质量检查，避免文字溢出、重叠和页面过密。
答辩时可以这样说：我们不是“模型直接生成一个 PPT 文件”，而是把课件生成拆成 课程理解、证据检索、课件规划、逐页生成、结构化编译、渲染验证、修复导出 七个受控环节。
